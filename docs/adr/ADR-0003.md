# ADR-0003: UI framework

- **Status:** Accepted  
- **Date:** 2026-07-09  
- **Deciders:** Phase 0 architecture (owner-approved)

## Context

Open question from the master guide: vanilla + canvas vs a lightweight framework (Preact/Svelte). The app needs dense forms (catalog, money, calendar), a canvas/WebGL house view, PWA potential, and agent-friendly code generation. Bundle size and maintainability both matter.

## Decision

Adopt **Vite + TypeScript + Preact** (`@preact/preset-vite`), with **preact-router** for navigation and **@preact/signals** available for local reactive state.

House-view plugins mount into a host `HTMLElement` and may use canvas APIs directly — they are not required to render via Preact VDOM.

## Consequences

### Positive

- Component model for command-center screens without React weight.
- Familiar JSX patterns for agent and human contributors.
- Clean split: Preact for app chrome; canvas for iso/pixel/3d plugins.
- Excellent Vite ecosystem (PWA plugin later, Pages deploy).

### Negative / trade-offs

- Smaller ecosystem than React (usually irrelevant with Preact compat if needed).
- Svelte fans may prefer different reactivity model — reversible at module boundaries if ever rewritten.

## Alternatives considered

| Option | Why not |
|--------|---------|
| Vanilla TS + canvas only | Too costly for multi-domain forms and routing |
| Svelte 5 | Excellent; Preact chosen for broader JSX familiarity in agent workflows |
| Full React | Larger runtime; no strong need |
| Next.js / SSR framework | Local-first SPA on Pages is simpler; SSR adds little for IDB app |
