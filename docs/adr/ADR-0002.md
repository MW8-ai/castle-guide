# ADR-0002: Default house-view renderer

- **Status:** Accepted (isometric default)  
- **Date:** 2026-07-09  
- **Deciders:** Phase 0 architecture (owner-approved)  
- **Amendment:** One throwaway iso spike required in Phase 0; pixel spike deferred

## Context

The owner wants a house view that is readable at a glance, supports drag-and-drop placement with real dimensions, appliance click-through to data cards, and broad appeal (GTA-hardened and cozy-game fans). Candidates:

1. Isometric sim (Sims / Two Point Hospital)  
2. Cozy retro pixel  
3. First-person 3D walkthrough  

Prime law: renderers are plugins over the home record; switching skins must lose no data.

## Decision

1. **Default renderer: isometric sim** (`iso`), full implementation in Phase 4.
2. **Phase 0 spike:** `spikes/iso-canvas` — two rooms from real dimension data, one clickable appliance placement — proves the plugin contract in real code. No polish.
3. **Retro pixel:** second skin in Phase 6 (or earlier if capacity).
4. **First-person 3D:** BLUEPRINT socket — data model stores `H` and optional `z` now; Three.js plugin later per HUMAN_DIRECTIONS. No Phase 0 pixel spike.
5. **Plugin contract** documented in ARCHITECTURE.md; host owns all mutations.

## Consequences

### Positive

- Best fit for layout readability + dimensioned drag-drop.
- Lower risk and accessibility burden than FPS walkthrough for core navigation.
- Broad demographic tolerance (strategy/sim literacy).
- 3D still possible without data rewrite.

### Negative / trade-offs

- Less "flex" than shipping 3D first.
- Needs minimal art strategy (procedural/SVG acceptable for v1).
- Iso projection math must stay consistent for placement coordinates.

## Alternatives considered

| Option | Why not default |
|--------|-----------------|
| Pixel-first | Charming but easier to read as "toy"; footprint drag still works but readability of multi-room homes suffers |
| 3D-first | Highest wow; higher a11y, asset, and performance risk for the core loop gate |
| No house view until late | Loses differentiator and demo script; contract-first + spike is cheaper insurance |

## Spike acceptance criteria

- [x] Two rooms rendered from `dims.L` × `dims.W`
- [x] One placement with footprint
- [x] Click fires `onSelectItem(itemId)`
- [x] No storage writes inside spike renderer

## Follow-ups

- Phase 4: production `src/houseview/iso` with drag-drop + health overlay.
- Phase 6: `pixel` skin; `walk3d` when human activates BLUEPRINT.
